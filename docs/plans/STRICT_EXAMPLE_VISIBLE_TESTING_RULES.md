# Strict Example Visible Testing Rules

Status: contract draft

Created: 2026-05-24

This file defines the testing rules for native examples. It exists because a
test that drives runtime events or a synthetic source-event shortcut can pass
while the visible app window is empty, stale, unfocused, or not interactive for
a human.

These rules apply to:

- `examples/todomvc.bn`
- `examples/cells.bn`
- every future example added to this repo

## Core Rule

An example is not verified as interactive until the verifier proves the same
visible native window that a user sees can receive input, update runtime state,
and redraw app-owned pixels.

Process existence is launch evidence only. It is not visual evidence and not
interaction evidence.

## Evidence Tiers

Reports must label evidence with one of these tiers:

- `runtime`: direct runtime calls such as `LiveRuntime::apply_source_event`.
- `host-synthetic`: synthesized `HostInputEvent` through the app input layer.
- `real-window`: events delivered to the exact visible native window/process.
- `human`: manual human observation.

Lower tiers must not satisfy higher-tier requirements. In particular:

- `runtime` must not count as UI interaction.
- `host-synthetic` must not claim raw human/OS interaction.
- `real-window` requires proof bound to the exact launched preview/dev PIDs.
- `human` reports require explicit human-observed artifacts and must not be
  generated by Codex/operator automation.

If a report has `real_os_input: false`, it must not claim the example is
human-interactive. It may only claim the lower evidence tier that actually ran.

## Exact Launch Binding

Visible launch verification must use the same launch form intended for the user:

```sh
cargo build -p boon_native_playground
cosmic-background-launch --workspace boon-circuit -- \
  ./target/debug/boon_native_playground --role desktop --example <example>
```

The report must bind all evidence to that launch:

- launcher PID
- desktop supervisor PID
- preview child PID
- dev child PID
- preview and dev window titles/tokens
- preview and dev role report paths
- source file path and hash
- binary hash
- artifact timestamps newer than source and binary
- app-owned framebuffer artifacts from those exact roles

Stale reports, stale screenshots, different verifier runs, and unrelated
desktop screenshots must fail the visible launch gate.

## App-Owned Pixel Proof

Whole-desktop screenshots are not proof for automated gates. The verifier must
capture app-owned framebuffer/readback artifacts from the preview and dev
windows.

For each visible launch, reports must include:

- preview framebuffer path and SHA-256
- dev framebuffer path and SHA-256
- image dimensions
- nonblank sample count
- unique color count
- clear/background ratio
- content bounding boxes
- structural text/control inventory

The gate must fail on:

- single-color windows
- debug palettes or rainbow placeholders
- large accidental blank regions
- stale app-owned frames
- screenshots not bound to launched PIDs

## Input Proof

Input verification must follow this path:

```text
real window event or synthetic HostInputEvent
-> app input adapter
-> focus/hit test
-> document source binding
-> runtime source dispatch
-> semantic delta
-> render patch
-> app-owned framebuffer change
```

The verifier must not skip directly from a test to
`LiveRuntime::apply_source_event` when claiming UI interaction.

Direct runtime calls are allowed only for the `runtime` tier and must be labeled
as such.

Every interaction scenario must record:

- target window PID and role
- target hit region
- focused element before and after
- input event sequence
- source binding resolved from the document
- runtime source ID/action evidence
- semantic delta summary
- render patch summary
- before/after framebuffer hashes or crop diffs
- whether the input was `real-window`, `host-synthetic`, or `runtime`

If the visible window cannot receive input, the gate must fail. It must not be
converted into a runtime-only pass.

## Example Manifest

Every example must have a manifest entry before it can be counted by an
all-examples readiness gate.

Suggested file:

```text
examples/manifest.toml
```

Each entry must define:

- example name
- source path
- required initial visible assertions
- required input scenarios
- required scroll/focus scenarios
- required visual artifacts
- required performance thresholds
- whether human testing is still needed

Future examples fail readiness until they add a manifest entry and pass all
generic native gates for that entry.


The old broad example-readiness command surfaces have been deleted. Native
readiness now flows through the manifest-backed native GPU gates in
`docs/architecture/native_gpu_handoff_manifest.json` and the aggregate
`verify-native-gpu-all --check-existing`. Do not restore a parallel
example-readiness path.

## TodoMVC Required Scenarios

Initial visible state must prove:

- preview title is `todos`
- initial demo rows are visible
- initial count is `3 items left`
- one initial completed row is visibly completed
- `Clear completed` visibility matches completed-count state
- input placeholder is `What needs to be done?`
- footer shows All, Active, and Completed filters
- dev window shows code editor and useful diagnostics/debug content
- neither window has large accidental blank areas

Interaction scenarios must prove:

- focus new-todo input
- type a new todo
- press Enter to add it
- reject empty/blank add
- toggle a single row
- toggle all rows
- switch All, Active, and Completed filters
- double-click a row to edit
- commit edit with Enter
- cancel edit with Escape
- commit edit on blur
- clear completed rows
- delete a row

Every scenario must prove runtime state changed when expected and preview pixels
changed when expected. The empty-add rejection is allowed to prove no runtime
or pixel change, but only if the report explicitly labels it as expected
no-op behavior.

## Cells Required Scenarios

Initial visible state must prove:

- grid is visible and not a placeholder
- row and column headers are visible
- initial demo data is visible
- empty cells display empty, not `0`
- focused cell style is visible
- focused cell right/bottom borders are visible
- formula/value input bar is visible
- formula/value input bar reflects the focused cell
- dev window shows code editor and useful diagnostics/debug content
- neither window has large accidental blank areas

Interaction scenarios must prove:

- click a cell and update focus
- type a scalar value
- commit with Enter
- edit a formula in the formula/value input bar
- commit a formula with Enter
- recompute dependent cells
- display errors for invalid/cyclic formulas
- cancel edit with Escape
- blur commits or cancels according to the spec
- vertical mouse-wheel scrolling changes visible rows
- Shift + mouse-wheel or horizontal wheel changes visible columns
- row headers and column headers remain aligned with the grid during scroll
- scrolling remains responsive for the full required row/column count

Every scenario must prove focus/hit-test/source-binding evidence, runtime state
evidence, render patch evidence, and before/after app-owned framebuffer
evidence.

## Performance Requirements

Performance checks must be per example and per window role.

Performance evidence must be honest. A performance gate must measure the exact
visible native app path or clearly label a lower evidence tier. It must not use
microbenchmarks, reduced example sizes, hidden headless paths, stale frames, or
runtime-only loops as proof that the visible app is fast.

Reports must include:

- preview first meaningful frame
- preview interaction latency p50/p95/p99
- preview frame time during repeated interactions
- dev editor scroll latency
- example-specific scroll latency where relevant
- sustained frame-rate evidence during scrolling and repeated edits
- dropped-frame count and longest-frame duration
- release-build and debug-build labels
- exact tested row/column/item counts
- exact launched preview/dev PIDs and window roles
- whether input was `real-window`, `host-synthetic`, or `runtime`
- allocation behavior after warmup
- evidence that dev/debug rendering does not slow preview interaction

The target for visible scrolling and common interactions is 60 FPS-class
behavior. Reports must include p50/p95/p99/max frame times and must fail when
p95 frame time exceeds the configured frame budget for the example. If a
machine-specific threshold is needed, it must be explicit in the report and
must not silently weaken the gate.

Cells must not pass by making the grid smaller than the required example size.
Performance fixes must preserve the required row/column count and interaction
surface.

Cells-specific speed requirements:

- measure wheel scrolling with many cells at the full required grid size;
- measure vertical and horizontal scrolling separately;
- include Shift-wheel or horizontal-wheel evidence;
- prove row/column headers stay aligned while scrolling;
- prove the formula bar/editor remains responsive while the grid is large;
- fail if the verifier cannot reproduce smooth visible scrolling;
- fail if a performance optimization hides cells, drops headers, disables hit
  testing, or reduces the required row/column count.

Dev-editor speed requirements:

- measure scrolling through the full source buffer, especially large examples;
- measure syntax highlighting and diagnostics update cost;
- fail if the visible editor clips the file, skips lines, or only renders a
  small static slice to look fast.

## Failure Rules

The verifier must fail when:

- the visible manual launch differs from automated artifacts
- an app appears empty when initial state is required
- visible controls do not match runtime state
- a control is visible when its Boon `visible` condition is false
- the user cannot interact with the visible window
- tests bypass focus/hit testing or document source bindings
- a source-event IPC shortcut is used as interaction proof
- a report claims real OS/human input without real OS/human evidence
- the tested process/window is not the same one shown to the user
- screenshots are stale or from a different launch

If a weaker tier passes but the required tier is missing, the report status must
be `fail` or `blocked`, not `pass`.

## Handoff Rule

After a successful automated run, the operator may start the visible app for
human confirmation. The final message must clearly say whether the evidence is:

- automated runtime only
- automated host-synthetic
- automated real-window
- human observed

Do not tell the user an example is visually or interactively correct unless the
current visible launch has been verified at the required tier.
